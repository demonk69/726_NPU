## 修复进展

### 1. Bug-20: FP16 Packer修复 ✅

**问题分析**:
- 原实现使用复杂的shift register组装两个16-bit半字
- 实际上PPBuf在FP16模式下直接输出完整的16-bit FP16值
- 不需要额外组装逻辑

**修复方案**:
- 完全移除FP16 packer的shift register逻辑
- 简化`pe_consume`为简单的`pe_en && (pe_data_ready || pe_flush)`
- 直接使用PPBuf输出作为PE输入

**代码修改** (`rtl/top/npu_top.v`):
```verilog
// 简化后的FP16逻辑
wire pe_data_ready = !w_ppb_buf_empty_int && !a_ppb_buf_empty_int;
wire pe_consume = pe_en && (pe_data_ready || pe_flush);
assign w_ppb_rd_en = pe_consume;
assign a_ppb_rd_en = pe_consume;

// 直接使用PPBuf输出
wire [DATA_W-1:0] pe_w_data = w_ppb_rd_data;
wire [DATA_W-1:0] pe_a_data = a_ppb_rd_data;
```

**修复后状态**:
- ✅ PE单元测试: 19/19 PASS
- ✅ FP16乘法器: 44/44 PASS
- ✅ FP16加法器: 20/20 PASS
- ⚠️ FP16 E2E: 3/9 PASS (OS模式工作正常，WS模式有问题)

### 2. Bug-21: WS模式问题分析 ✅

**问题现象**:
- T2, T4, T6 (WS K=1) 全部得到0，期望3.0, -6.0, 1.0
- T1, T3, T5 (OS K>1) 全部PASS

**根本原因分析**:
1. **OS模式工作正常**：T1, T3, T5都是OS模式，PASS
2. **PE单元测试WS模式正常**：直接驱动PE时WS工作正常
3. **问题在系统集成层**：涉及PPBuf、DMA、FIFO的交互

**关键发现**:
- INT8综合测试8/8 PASS，但**全是OS模式**
- PPBuf在FP16模式下K=1时，fill count = 2（不是1）
- 导致2个pe_consume：第一个产生正确结果，第二个产生0
- DMA只读1个结果，但可能读到了第二个（0）

**待修复**：需要深入分析FIFO写入顺序
